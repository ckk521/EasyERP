# 入库管理模块 - 用户故事

> 模块说明: 管理商品入库全流程，支持采购预约入库和直接入库两种渠道
> 优先级: P0 (MVP必需)
> 更新日期: 2026-04-21

---

## 一、核心设计原则

### 1.1 数据关联原则

**所有业务操作必须有数据源，不能凭空创建：**

```
入库单 ──产生──▶ 入库明细 ──产生──▶ 库存记录
   │                │
   │                └──验收不合格──▶ 退货记录
   │                                    │
   │                                    └──关联原入库明细
   │
   └──产生──▶ 库存事务记录

示例:
入库单IN001 ──包含──▶ SKU-001 x 100
                    ├── 验收合格 80件 ──上架──▶ 库存记录
                    └── 验收不合格 20件 ──退货──▶ 退货记录(关联IN001)
```

### 1.2 数据追溯链

```
查询某个商品的完整链路:

SKU-001 库存 80件
├── 来源: 入库单 IN202604210001
├── 批次: IN202604210001-0001
├── 供应商: 深圳电子科技
├── 送货批次: SZ-0421-AM
├── 入库时间: 2026-04-21 14:30
├── 验收人: 张三
├── 上架人: 李四
└── 库位: ZN-A-01-02-03

退货记录 20件
├── 来源: 入库单 IN202604210001 (同一批货)
├── 退货原因: 包装破损
├── 退货时间: 2026-04-21 15:00
└── 处理方式: 退货供应商
```

## 二、单据编号规则

### 2.1 编号格式

```
格式: 前缀 + 年月日 + 4位序号

单据类型          前缀    示例           说明
─────────────────────────────────────────────────────
入库单 (Inbound)   IN     IN202604210001  Inbound
退货单 (Return)    RT     RT202604210001  Return
出库单 (Outbound)  OUT    OUT202604210001 Outbound
调拨单 (Transfer)  TR     TR202604210001  Transfer
盘点单 (Stocktake) ST     ST202604210001  Stocktake
库存事务 (Trans)   TX     TX202604210001  Transaction
```

### 2.2 批次号规则

```
格式: 入库单号 + SKU后4位 + 2位序号
示例: IN202604210001-0001-01

生成时机: 验收通过后自动生成
```

### 3.2 两种入库方式

| 入库类型 | 来源 | 有PO号 | 有供应商 | 适用场景 |
|---------|------|--------|---------|---------|
| **采购入库** | ERP推送采购订单 | ✓ | ✓ | 正常采购到货 |
| **直接入库** | WMS手动创建 | ✗ | 可选 | 退货/调拨/赠品/盘盈 |

## 四、入库单管理

### Story 2.1 采购预约入库单接收
**作为** 仓库收货员  
**我希望** 能够接收ERP推送的采购预约入库单  
**以便** 提前知晓到货信息，做好收货准备

**验收标准:**
- [ ] 自动接收ERP采购订单推送，生成待收货状态的入库单
- [ ] 入库单显示采购订单号（PO号）、供应商信息、预计到货日期
- [ ] 入库单明细显示商品SKU、名称、预期数量
- [ ] 可以查看入库单的生命周期状态（待收货→收货中→验收中→待上架→已完成）
- [ ] 支持按采购订单号、供应商、状态筛选查询
- [ ] 界面标识显示"采购预约"标签（蓝色）

---

### Story 2.2 直接入库单创建
**作为** 仓库管理员  
**我希望** 能够在WMS直接创建入库单  
**以便** 处理退货入库、调拨入库、盘盈入库等非采购入库场景

**验收标准:**
- [ ] 可以手动创建入库单，选择入库类型
- [ ] 支持入库类型：退货入库/调拨入库/盘盈入库/赠品入库/其他入库
- [ ] 直接入库单无采购订单号，但可以选择供应商（用于模拟采购入库场景）
- [ ] 可以添加入库商品明细（SKU、数量）
- [ ] 可以保存草稿或提交（提交后状态变为待收货）
- [ ] 界面标识显示"直接入库"标签（灰色）

---

### Story 2.3 送货批次号管理
**作为** 仓库管理员  
**我希望** 能够为入库单设置送货批次号  
**以便** 区分同一供应商同一天的多批送货

**验收标准:**
- [ ] 创建入库单时可以填写送货批次号（如：SZ-0421-AM）
- [ ] 送货批次号用于区分同一供应商同一天的不同送货批次
- [ ] 支持按送货批次号查询入库单
- [ ] 入库单详情显示送货批次号

**示例:**
```
供应商: 深圳电子科技
├── 送货批次 SZ-0421-AM (上午) → 入库单 IN202604210001
└── 送货批次 SZ-0421-PM (下午) → 入库单 IN202604210002
```

---

### Story 2.4 入库单编辑与取消
**作为** 仓库管理员  
**我希望** 能够编辑或取消未完成的入库单  
**以便** 处理信息变更或异常情况

**验收标准:**
- [ ] 待收货状态的入库单可以编辑商品明细
- [ ] 待收货状态的入库单可以取消（需填写取消原因）
- [ ] 已开始收货（收货中及以上状态）的入库单不可编辑
- [ ] 已取消的入库单可以重新激活或复制创建新单

## 五、收货作业

### Story 2.5 收货登记
**作为** 仓库收货员  
**我希望** 能够对到货商品进行收货登记  
**以便** 确认实际到货数量和状态

**验收标准:**
- [ ] 扫描入库单号或采购订单号定位到待收货单据
- [ ] 扫描商品条码/SKU码自动识别商品
- [ ] 录入实际收货数量
- [ ] 支持部分收货（分批到货场景）
- [ ] 收货完成后提交，状态变为"收货中"
- [ ] 全部商品收货完成后状态变为"验收中"

---

### Story 2.6 收货数量差异处理
**作为** 仓库收货员  
**我希望** 能够在收货时处理数量差异  
**以便** 准确记录实际到货数量

**差异类型:**
- 少货: 实际数量 < 预期数量
- 多货: 实际数量 > 预期数量
- 包装差异: 整箱数量与标注不符

**验收标准:**
- [ ] 收货时可以录入实际到货数量
- [ ] 系统自动计算差异数量
- [ ] 差异超过阈值时需要确认原因
- [ ] 少货需要拍照记录现状
- [ ] 可以选择是否部分收货
- [ ] 差异记录自动关联入库单

**差异处理选项:**
```
少货处理选项:
├── 确认收货（按实际数量）
├── 拒收差异部分（只收预期数量）
├── 全部拒收（货物与单据差异过大）
└── 暂存待处理（等待供应商确认）

多货处理选项:
├── 确认收货（按实际数量）
├── 拒收多出部分
└── 暂存待处理（等待供应商确认）
```

---

### Story 2.7 收货商品异常处理
**作为** 仓库收货员  
**我希望** 能够在收货时发现商品异常  
**以便** 及时记录和处理

**异常类型:**
- 包装破损: 外包装损坏
- 商品损坏: 商品本体损坏
- 错货: 收到的商品与单据不符
- 串货: 同一SKU不同批次/版本混在一起

**验收标准:**
- [ ] 收货时可以选择商品异常标记
- [ ] 异常需要选择具体类型（包装破损/商品损坏/错货/串货）
- [ ] 需要拍照上传异常照片（最多5张）
- [ ] 需要录入异常描述
- [ ] 异常商品与正常商品分开处理
- [ ] 可以选择是否拒收异常商品
- [ ] 异常记录自动关联入库单和供应商

## 六、验收作业

### Story 2.8 质量验收
**作为** 仓库质检员  
**我希望** 能够对收货商品进行质量验收  
**以便** 确保入库商品质量合格

**验收标准:**
- [ ] 验收列表显示待验收商品
- [ ] 可以逐件或批量验收
- [ ] 验收结果：合格/不合格
- [ ] 不合格需填写原因：破损/过期/规格不符/质量问题/其他
- [ ] 不合格商品可以拍照记录
- [ ] 效期管理商品必须录入生产日期和过期日期
- [ ] 支持部分验收（分批验收场景）
- [ ] 验收完成后状态变为"待上架"

**效期商品验收:**
- 必须录入：生产日期、过期日期
- 系统校验：过期日期 ≥ 当前日期 + 最小保质期
- 临期商品系统自动标记预警

---

### Story 2.9 不合格品处理
**作为** 仓库主管  
**我希望** 能够处理不合格品  
**以便** 确定最终处理方式

**处理方案:**

| 处理方式 | 适用场景 | 库存处理 | 财务影响 |
|---------|---------|---------|---------|
| 退货供应商 | 供应商责任 | 移到退货区 | 向供应商索赔 |
| 降价销售 | 轻微瑕疵不影响使用 | 移到降价区 | 损失差价 |
| 报损 | 无法销售 | 移到报损区 | 计入损失 |
| 内部使用 | 员工自用/样品 | 内部领用 | 成本内部消化 |
| 销毁 | 过期/有毒有害 | 实物销毁 | 计入损失 |

**验收标准:**
- [ ] 可以查看所有不合格品记录
- [ ] 不合格品记录关联原入库单和入库明细（数据源）
- [ ] 可以选择处理方式
- [ ] 需要录入处理数量
- [ ] 需要审批的处理方式（金额/数量阈值）需要主管审批
- [ ] 审批通过后自动生成退货记录（如选择退货供应商）
- [ ] 生成财务需要的处理记录

## 七、上架作业

### Story 2.10 智能库位推荐
**作为** 仓库上架员  
**我希望** 系统能够推荐合适的库位  
**以便** 提高上架效率并优化库位利用

**验收标准:**
- [ ] 系统根据商品属性自动推荐库位：
  - 温度要求匹配（常温/冷藏/冷冻）
  - 安全等级匹配（普通/高值/危险品）
  - 库位尺寸限制匹配
  - 同SKU集中存放（如策略允许）
- [ ] 推荐库位按优先级排序（近出库口优先）
- [ ] 可以手动选择其他库位
- [ ] 推荐库位已满时给出提示

**库位分配规则:**
```
IF 商品.存储条件 = "冷藏" THEN 推荐 冷藏库位
ELSE IF 商品.存储条件 = "冷冻" THEN 推荐 冷冻库位
ELSE IF 商品.是否高价值 = true THEN 推荐 高值库位
ELSE 推荐 常温标准位
```

## 九、数据关联约束

### 9.1 强制关联规则

```
操作              必须关联的数据源
─────────────────────────────────────────────────────────
收货              入库单(已创建)
验收              入库明细(已收货)
上架              入库明细(已验收合格)
退货              入库明细(验收不合格)
库存变动          上架记录/退货记录
调拨              库存记录(已存在)
出库              库存记录(已存在)
```

### 9.2 数据完整性校验

```
入库单完成时校验:
总预期数量 = 总收货数量 + 收货差异数量
总收货数量 = 总合格数量 + 总不合格数量
总合格数量 = 总上架数量 + 待上架数量
总不合格数量 = 总退货数量 + 其他处理数量

库存校验:
库存数量 = SUM(上架记录.上架数量) - SUM(出库记录.出库数量)
```
